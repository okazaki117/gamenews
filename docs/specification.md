# ゲームビジネスニュース自動要約・GitHub Wiki更新システム 仕様書

## 1. システム概要
RSSフィードからゲームビジネス・業界関連のニュースを自動取得し、LLM（DeepSeek / Gemini API）を用いて要約を生成。その結果を対象プロジェクトの GitHub Wiki リポジトリ (`.wiki.git`) へ自動的に投稿・更新するシステムです。
ユーザーが朝8時ちょうどに記事を確認できるよう、GitHub Actionsを利用して毎日午前7時50分（日本時間）に完全自動で実行される日次バッチ処理として構築します。

## 2. システムアーキテクチャ
- **実行基盤**: GitHub Actions (Ubuntu)
- **開発言語**: Python 3.10 以降
- **AI/LLM連携**:
  - メイン: DeepSeek API (高品質な文章要約)
  - サブ/代替: Gemini API
- **永続化データ層**: GitHub Wiki (Markdownファイル管理のGitリポジトリ)
- **インフラ構成の思想**: サーバーレス、ステートレス。状態を持たず、実行時の日付情報を基準に新規ページの作成や更新を判断します。

## 3. データフロー
1. **トリガーの起動**: GitHub Actionsがスケジュール（Cron: `50 22 * * *` = UTC前日22時50分/JST7時50分）により起動。
2. **フィードの取得**: `config/feeds.json` に設定されたRSSフィードURL一覧から最新記事の一覧を取得。
3. **記事のフィルタリングと解析**: 直近24時間以内に公開された未処理ニュース記事を抽出し、HTMLタグ除去などのクレンジングを実施。
4. **要約の生成 (LLM)**: クレンジングされた記事本文（またはdescription）とシステムプロンプトをLLM APIに送信し、「個別記事の日本語による要約（元の英語表題の日本語訳、概要、箇条書き）」を取得。
5. **全体トレンドの要約 (LLM)**: 全ての記事の個別要約テキストを統合して再度LLM APIへ送信し、「今日のゲーム業界全体のトレンドや重要なニューストピック」を3行程度の全体サマリーとして生成。
6. **Markdownコンテンツ生成**: 「全体サマリー」をWikiページの冒頭に配置し、その後に統合された「各記事の個別要約」を配置し、1つのMarkdownページ（例: `GameBizNews-YYYY-MM-DD.md`）としてテキストを構築。
7. **Git操作 (Wiki更新)**:
   - システム実行時に `.wiki.git` リポジトリをローカルディレクトリへチェックアウト。
   - 生成したMarkdownファイルを配置。
   - `git add`, `git commit`, `git push` を経由してリモートのWikiリポジトリを更新。

ルートディレクトリ (リポジトリルート)
```text
.
├── .github/
```

## 5. 使用ライブラリ（Python）
- **`feedparser`**: RSS/Atomフィードのパース、データの構造化に利用。
- **`requests` / `beautifulsoup4`**: HTTP通信処理と、RSS本文のHTMLタグ除去（スクレイピング）に利用。
- **`openai`** (または `requests`): DeepSeek APIは通常OpenAI互換のREST APIを提供しているため、OpenAIのPythonクライアントを利用して手軽に呼び出すことが可能。
- **`google-generativeai`**: (Geminiを使用する場合)
- **`pytz` / `datetime`**: JSTの正確な時刻取得と、日付を基準としたファイル命名に利用。

## 6. セキュリティと機密情報の取り扱い
- APIキー(DeepSeek/Gemini)やGitHubトークン(`GITHUB_TOKEN` 等)等のクレデンシャル情報は、**いかなる場合もコード上にハードコードしない**こと。
- ロジック内では常に `os.environ.get('VAR_NAME')` を用いて、環境変数から動的に読み込む設計とする。
- 実行時のコンソールログにおいて、APIキーや長大な生データをダンプしないこと。

## 7. エラーハンドリング指針
- **RSS処理エラー**: 一部のフィードURLがダウンしていたり取得に失敗した場合でも、システム全体をクラッシュさせず、エラーログのみを出力して次のフィードの処理へ継続させる（スキップ処理）。
- **API通信エラー**: LLM APIのレート制限（HTTP 429）やサーバーエラー（HTTP 500系）の発生時、リトライ機構を組み込む。また、最終的に失敗した場合でも処理対象スキップとし、システムを止めない。
- **Git操作競合**: `git push` 時にリモート側の変更によって競合（コンフリクト）が発生しないよう運用する想定だが、万が一の失敗時は、Actions上で失敗ログとして記録・通知させる。
