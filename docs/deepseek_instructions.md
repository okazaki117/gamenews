# 実装担当AI（DeepSeek）向け 詳細開発指示プロンプト集

あなたは「実装特化のプログラマーAI」です。
以下の【システム基本情報】と【共通ルール】を理解した上で、各「フェーズ」に分割された指示に基づいて、具体的なPythonコードおよび設定ファイルの生成を行ってください。

## 【システム基本情報】
- **目的**: RSSからゲームビジネスニュースを取得し、LLMによって要約した上で、対象プロジェクトの GitHub Wiki (`.wiki.git`) に毎日自動投稿する。
- **実行基盤**: GitHub Actions (Ubuntu環境)
- **実行頻度**: 毎日 JST 07:50 の日次バッチ実行（ユーザーが8時に確認できるようにするための前倒し）。
- **使用言語・主要ライブラリ**: Python 3.10以降, `feedparser`, `requests`, `beautifulsoup4`, `openai` (DeepSeek用) または互換クライアント。

## 【共通（厳守）ルール】
1. **言語設定**: コード内のコメント、関数やクラスのdocstring（ドキュメンテーション文字列）、および出力するテキスト情報（要約、ログメッセージなど）は、**すべて日本語**で記述してください。
2. **命名規則**: 変数名、関数名、ファイル名などは明確な「英語」で命名すること（ローマ字命名は絶対に禁止）。
3. **機密情報の排除**: APIキーやGitHubトークンはコード内にハードコードせず、必ず環境変数（`os.environ`等）から読み込む設計にしてください。
4. **エラーハンドリング**: ループ処理の途中で一部の記事やフィードでエラー（ネットワークエラーやパース失敗等）が発生してもプログラム全体がクラッシュしないよう、適切な `try-except` 句を設けてエラーログを標準出力し、次の処理へ継続（continue）するように実装してください。
5. **過去のバグからの必読アンチパターン**: 過去の実装で「Windows固有の削除コマンド(rmdir等)の使用」「GitHub連携URLでの x-access-token 指定漏れ」「Pythonのバッファ落ちによるログ消失」などの不具合が発生しました。実装の際は必ず `docs/AI_IMPLEMENTATION_GUIDELINES.md` を事前に熟読し、システムを破壊するのと同じアンチパターンを絶対に繰り返さないよう細心の注意を払ってください。

## 開発フェーズと指示書

### 【フェーズ1】環境構築とニュース取得・要約ロジックの実装
**指示内容:**
以下の要件に基づくPythonスクリプトおよび設定ファイル群（`config/feeds.json`, `requirements.txt`, `src/rss_parser.py`, `src/summarizer.py`）を実装してください。

1. **`config/feeds.json`**
   - ゲームニュースサイトのRSSフィードURLを2〜3個設定したJSONファイルを作成する。（URLはダミーまたは実在の汎用的なものでも可）
2. **`requirements.txt`**
   - 本プロジェクトに必要なライブラリ名をリストアップする（バージョン指定は不要、最新想定）。
3. **`src/rss_parser.py`**
   - JSONファイルからURLリストを読み込み、`feedparser` を用いてRSSを解析するモジュールを作成する。
   - 「記事の公開日時が『実行日時の直近24時間以内』であること」を条件に絞り込むロジックを含める。
   - 本文（description）が含まれる場合、`beautifulsoup4` を活用して不要なHTMLタグを除去したプレーンテキストとして抽出する。
   - 抽出結果として、タイトル、URL、プレーンテキストのリストを返却する機能とする。
4. **`src/summarizer.py`**
   - 環境変数 `LLM_API_KEY` を取得し、DeepSeek API経由で記事本文を要約する機能を作成する。（OpenAI互換ライブラリを用いて、base_urlをDeepSeekのアドレスに変更する実装を推奨）
   - システムプロンプトを用いて「この記事の重要なポイントを3〜5行程度の箇条書きで、日本語により分かりやすく要約してください」と指示を与えること。
   - API通信エラー時は `None` や空文字を返す、またはリトライを行うエラートラップを設けること。

### 【フェーズ2】GitHub WikiへのGit操作とメインロジック統合
**指示内容:**
フェーズ1で作成したモジュールと連携し、要約データをWiki用に整形してGitリポジトリに保存・自動コミットする処理群（`src/wiki_updater.py`、`src/main.py`）を実装してください。

1. **`src/wiki_updater.py`**
   - 受け取った要約データリストをもとに、当日の日付を用いたMarkdownファイル（例: `GameNews_YYYYMMDD.md`）をフォーマットして生成するロジックを作成する。
   - Pythonの `subprocess` モジュール等を用いて、指定されたローカルディレクトリ（Wikiリポジトリのパス）内で以下のGitコマンドを連続実行する関数を作成する。
     1. `git config` を用いたボット用ユーザー名・メールアドレスの設定
     2. `git add .`
     3. `git commit -m "docs: Auto update daily game news (YYYY-MM-DD)"`
     4. `git push`
   - Git操作に関するエラー時は適切に例外をキャッチし、ログを出力すること。
2. **`src/main.py`**
   - 全体を統合するエントリーポイントとして作成する。
   - 「RSSの取得」→「AIによる要約生成」→「Markdown化とGit Wikiへのプッシュ」という一連のパイプラインを実行する構成にする。
   - 実行経過や処理進捗がログ（標準出力 または `logging`）として分かるように出力処理を組み込む。

### 【フェーズ3】GitHub ActionsのCI/CD設定（自動実行設定）
**指示内容:**
本システムを毎日自動で実行させるための GitHub Actions ワークフロー定義ファイル（`.github/workflows/update_wiki.yml`）を作成してください。

**要件:**
1. **トリガーの定義**:
   - `schedule`: Cron式を用いて「毎日 午前9時 (日本標準時: JST)」に実行されるように設定する（UTC換算で記述すること）。
   - 手動でも実行・テストできるように `workflow_dispatch` トリガーも併記する。
2. **実行されるジョブ (Job) の定義**:
   - `ubuntu-latest` を実行環境とする。
   - `actions/checkout` を用いて、まずメインのソースコードリポジトリをチェックアウトする。
   - その後、ステップを分けてWiki側のリポジトリ（`.wiki.git`）を専用のディレクトリへチェックアウトする設定を含める。
   - Python環境（バージョンに注意）をセットアップし、`requirements.txt` からライブラリをインストールする。
   - 最後に `python src/main.py` を実行する。
3. **環境変数 (env) の設定**:
   - スクリプト実行ステップにおいて、GitHub Secretsから `LLM_API_KEY` を渡すように記述する。
   - Wikiリポジトリのクローン・プッシュに必要な権限情報として、GitHubトークンが利用できる構成にすること。
